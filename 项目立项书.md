| 项目 | 内容 |
|---|---|
| 文档状态 | Draft |
| 日期 | 2026-05-15 |
| 项目名称 | BabyLog |
| 产品形态 | 阶段一 Android 原生 APK（主线）；阶段二再评估 PWA / iOS 走向 |
| 当前阶段 | 阶段一单机 MVP（Android 原生），预留后端同步和 OCR 接口；PWA 实现 (`app/`) 暂搁置 |

## Scope

BabyLog MUST 覆盖从孕期全阶段到宝宝 3 岁的家庭记录场景。

**阶段一**：MUST 先做 Android 原生 APK（`android-native/`，Java + Android SDK，无第三方依赖），在小米真机上跑通；数据 MUST 存储在本机（当前 `SharedPreferences`，计划迁 Room）。阶段一 MUST 在没有服务器、没有账号体系的情况下完整可用。

**阶段二**：在阶段一稳定使用后再评估是否同时维护 PWA / iOS 版本；`app/` 下的 React PWA 实现作为数据模型与 UI 参考保留。

跨端约束（iOS Safari / Android Chrome 通用）原属阶段二目标，阶段一以 Android 原生为单一目标。

当前 App MUST 仅支持单胎/单宝宝。`Family` 与 `ChildProfile` 在数据模型层保持 1:N 关系以预留多胎扩展，但首页、时间线、表单、曲线、设置 MUST NOT 出现"切换宝宝/选择宝宝"入口。多胎/双胎 UI 推迟到第一版之后再评估。

BabyLog 的第一目标用户 MUST 是项目发起人自己的家庭。产品设计 SHOULD 优先保证自家长期稳定可用、隐私可控、数据可导出。后续如果推荐给同事家庭使用，系统 MAY 扩展为小范围邀请制，但 MUST NOT 在第一版按公开商业产品、社区产品或大规模 SaaS 设计。

第一版 MUST 同时支持 iOS Safari / Home Screen Web App 和 Android Chrome / installed PWA。功能设计 MUST 优先使用两端都可用的 Web 标准；如果某能力只在单端可用，MUST 提供跨端降级路径。

第一版 MUST 预留同步、附件上传、B 超单识别的接口边界，但 MUST NOT 依赖服务器才能运行。后端地址、地域、证书和部署方式均 MUST 作为配置项保留，不得写死到源码或公开文档。B 超单识别 SHOULD 优先使用用户本机手动配置的 OpenAI-compatible 多模态模型 API，API Key MUST 只保存在本机，不得通过家庭服务器下发、同步或备份。

## Terms

| 术语 | 含义 |
|---|---|
| PWA | Progressive Web App，可添加到 iPhone / Android 主屏幕 |
| 单机版 | 仅使用本地 IndexedDB 和本地附件缓存，不依赖后端 |
| 后端预留 | 前端保留 API client、adapter、baseUrl 配置和错误状态，但当前实现为 mock/no-op |
| 家庭空间 | 多个照护人共享的同一个宝宝数据空间，第一版以本地概念保留 |
| 小范围推荐 | 后续 MAY 支持给同事家庭使用的邀请制部署或多家庭隔离，但不是 MVP 必需能力 |
| 孕期记录 | 预产期、孕周、产检、B 超、胎动、宫缩、孕妈指标等 |
| 育儿记录 | 喂养、睡眠、尿布、体温、用药、生病、成长数据等 |
| FGR | 胎儿生长受限，仅做参考提示，不做诊断结论 |
| OCR | 从 B 超单照片中识别检查日期、BPD、HC、AC、FL、EFW，以及胎心率、羊水、胎盘、胎位、宫颈管、CRL/NT 等公共字段；不识别或推断孕周 |

## Project Goals

| 目标 | 要求 |
|---|---|
| 全阶段时间线 | MUST 支持孕期、出生、0-3 岁连续记录 |
| 快速记录 | MUST 支持手机单手快速新增常用记录 |
| 成长曲线 | MUST 支持胎儿发育曲线和出生后身高、体重、头围曲线 |
| B 超单归档 | MUST 支持拍照保存 B 超单，SHOULD 支持后续 OCR 自动识别 |
| 多端同步预留 | MUST 抽象本地仓库、同步适配器和 OCR 适配器 |
| 数据归属 | MUST 支持本地导出，后续 SHOULD 支持自托管云端备份 |
| 跨端一致性 | MUST 在 iOS 和 Android 上提供同一套核心记录能力；验证顺序 MUST 以 iOS 为先，Android 紧跟回归 |
| 小范围扩展 | SHOULD 预留多家庭隔离和邀请制账号模型，但 MUST 不影响单家庭单机 MVP |

## User Scenarios

| 场景 | 用户动作 | 系统响应 |
|---|---|---|
| 孕期产检后记录 | 拍 B 超单，录入 BPD/HC/AC/FL/EFW | 保存照片和结构化数据，更新胎儿曲线 |
| 晚孕期观察胎动 | 打开胎动计数器，记录一段胎动 | 保存计数会话和趋势 |
| 分娩前记录宫缩 | 使用宫缩计时器 | 记录每次宫缩开始、结束、间隔 |
| 出生后夜间喂奶 | 点快捷按钮记录母乳/奶瓶 | 更新时间线和今日统计 |
| 儿保前查看资料 | 打开成长曲线和最近记录 | 显示身高体重头围趋势、喂养睡眠摘要 |
| 更换手机或备份 | 导出 JSON | 生成完整备份文件 |

## Record Time Semantics

| Item | Requirement |
|---|---|
| Default day boundary | MUST use local natural day `00:00` as the default summary boundary |
| User override | SHOULD allow changing day boundary to `04:00` for night-feeding families |
| Event time | MUST store `occurredAt` as an ISO timestamp with local timezone context |
| Summary time | MUST calculate "today" from the configured day boundary, not from UTC date |

### Deferred Until Backend Is Ready

| 模块 | 延后原因 |
|---|---|
| 多成员实时同步 | 需要服务器账号、鉴权、冲突合并 |
| 多家庭邀请制 | 需要后端账号、家庭隔离、邀请链接和权限模型 |
| 托管 OCR 服务 | 第一版不走 BabyLog 后端；使用用户本机配置的多模态模型 API 生成候选字段 |

| Web Push 提醒 | 需要 HTTPS、推送服务和订阅管理 |

### Common Entity Fields

| Field | Type | Required | Rule |
|---|---|---|---|
| id | string | Yes | MUST be globally unique client-generated UUID |
| familyId | string | Yes for family-scoped entities | MUST be present even in single-family MVP |
| childId | string | Yes for child-scoped entities | MUST point to the current single child in MVP |
| createdAt | ISO datetime | Yes | MUST be immutable after creation |
| updatedAt | ISO datetime | Yes | MUST update on every local edit |
| updatedBy | string | Yes | MUST be `"local"` in single-device MVP; future backend MAY replace with userId |
| schemaVersion | integer | Yes | MUST support future migrations |
| deletedAt | ISO datetime or null | Yes | MUST be null for active records; non-null means tombstone |

### Sync Conflict Strategy

| Item | Requirement |
|---|---|
| MVP strategy | MUST use entity-level last-write-wins based on `updatedAt` within the same `familyId` |
| Delete strategy | MUST preserve tombstones through `deletedAt` until a future compaction step |
| Scope | MUST never merge records across different `familyId` values |
| Logical clock | MUST NOT introduce vector clocks in MVP; MAY add a monotonic local revision later if conflict evidence requires it |

## Architecture

```text
PWA UI
  -> Application Services
      -> LocalRepository
          -> IndexedDB
      -> AttachmentRepository
          -> IndexedDB Blob / Browser storage
          -> Image compression before save
      -> SyncAdapter
          -> MockSyncAdapter now
          -> TencentCloudApiAdapter later
      -> OcrAdapter
          -> ManualOnlyOcrAdapter now
          -> PaddleOcrApiAdapter later
```

### Attachment Pipeline

| Step | Requirement |
|---|---|
| Image import | MUST support camera capture and photo library selection |
| Compression | MUST compress images before local storage by default |
| Default target | SHOULD use longest edge <= 2048 px and JPEG quality 0.8 for generated copies |
| Original file | MAY allow preserving original files, but MUST show storage impact before enabling |
| Quota display | MUST expose `navigator.storage.estimate()` usage in Settings when available |
| Per-record limit | SHOULD limit ultrasound attachments to 3 images per exam in MVP |

## API Contract Reserved

### Backend Availability Assumption

| Item | Requirement |
|---|---|
| Current backend state | MUST assume unavailable |
| Server provider | SHOULD assume Tencent Cloud self-hosted backend |
| Region | MUST be configurable; current purchased node may be Tokyo or another overseas region |
| Base URL | MUST be empty or disabled by default in single-device MVP |
| Runtime behavior | MUST keep all local features working when base URL is unset or unreachable |
| Health check | SHOULD be implemented as a future `/health` endpoint before enabling sync |
| Transport | Future backend MUST use HTTPS for camera upload, sync, and PWA production use |
| Fallback | Any failed backend call MUST leave local data intact and mark changes as `pendingSync` |

### Local Interfaces

| Interface | Required Methods | Current Implementation | Future Implementation |
|---|---|---|---|
| LocalRepository | list, get, put, delete, queryByDate | IndexedDB | IndexedDB |
| SyncAdapter | configure, healthCheck, pushLocalChanges, pullRemoteChanges, resolveConflict | Mock no-op | Tencent Cloud API |
| SmartVisionAdapter | recognizeUltrasoundImage | User-configured multimodal model | OpenAI-compatible multimodal API |
| AttachmentStore | saveBlob, getBlob, deleteBlob | IndexedDB Blob | Local + cloud object storage |

### Future Backend Endpoints

| Method | Path | Purpose | Notes |
|---|---|---|---|
| GET | /health | 检查后端可用性 | 后端接入前必须先实现 |
| POST | /auth/login | 登录 | 后端阶段启用 |
| GET | /families/:id/snapshot | 拉取家庭数据快照 | 返回完整实体和版本 |
| POST | /families/:id/changes | 上传本地变更 | 需要幂等 changeId |
| POST | /attachments | 上传附件 | 返回 attachmentId 和 URL |
| GET | /attachments/:id | 下载附件 | 需要权限校验 |
| - | 本机多模态识别 | 不经过 BabyLog 后端 | 用户点击后由本机调用已配置模型 API，返回候选字段 |

## UI Direction

The UI SHOULD be calm, medical-record-like, and usable at night with one hand. It MUST avoid heavy decoration, excessive pink styling, and dense marketing content.

| Area | Direction |
|---|---|
| Visual tone | 温和、干净、低干扰 |
| Primary color | 蓝绿或青绿色 |
| Layout | 底部导航 + 快捷记录按钮 + 时间线 |
| Density | 首页高信息密度，表单分步输入 |
| Accessibility | 大触控区域、夜间可读、低亮度模式 |

## Risks

| Risk | Impact | Mitigation |
|---|---|---|
| 医疗误用 | 用户可能把参考提示当诊断 | MUST 显示“仅供记录和参考，不能替代医生判断” |
| OCR 错误 | B 超指标识别错误影响曲线 | MUST 人工确认后入库 |
| 单位混淆 | cm/mm、kg/g 容易错 | MUST 在录入和确认页显示单位并做范围校验 |
| IndexedDB 数据丢失 | 浏览器清理可能删除数据 | MUST 提供 JSON 导出；SHOULD 提醒定期备份 |
| iOS PWA 限制 | 后台提醒、存储、安装态行为有限 | MUST 引导添加到主屏幕；MUST 在可用时请求持久化存储；推送后置 |
| 附件容量耗尽 | B 超单、证件和疫苗本图片可能耗尽浏览器配额 | MUST 默认压缩图片；MUST 显示存储用量；SHOULD 限制单记录附件数量 |
| Android/iOS 差异 | 相机、安装、缓存、后台能力表现不一致 | MUST 优先使用跨端 Web 标准，并在两端做手动验证 |
| 腾讯云暂未连通 | 同步、云附件不可用 | MUST 以 mock adapter 运行，后端配置为空时不得报阻塞错误；B 超识别与家庭服务器解耦 |
| 跨境医疗信息合规 | 腾讯云海外节点可能涉及敏感个人信息跨境传输 | MUST 在启用同步前做单独确认；MUST 显示后端地域；SHOULD 优先选择境内节点或自家明确知情配置 |
| 同步冲突 | 后续多端写入可能覆盖记录 | MUST 在 MVP 预置版本、更新时间和 tombstone 字段；MUST 使用 familyId 隔离 |
| 小范围分享后的隐私责任 | 同事家庭数据不应混入自家数据 | 后端阶段 MUST 实现家庭级数据隔离和账号权限；未实现前不开放多人多家庭使用 |
| 曲线数据授权 | 部分曲线表格可能有版权限制 | MUST 记录来源和许可；不可用时只保留适配接口 |

| M6 同步接入 | 后端适配器 | 腾讯云服务准备后执行 |
| M7 小范围邀请 | 多家庭隔离、邀请制账号 | 仅在自家使用稳定后执行 |

## Acceptance Criteria

| ID | Criteria |
|---|---|
| AC-01 | App MUST run locally in browser and remain usable after refresh. |
| AC-02 | App MUST work without backend for all MVP records. |
| AC-03 | User MUST be able to export all structured data as JSON. |
| AC-04 | User MUST be able to attach a B 超单 photo to an ultrasound record. |
| AC-05 | Growth charts MUST render from manually entered data. |
| AC-06 | Sync and OCR MUST be replaceable through adapters without rewriting UI forms. |
| AC-07 | Medical reference hints MUST NOT present themselves as diagnosis. |
| AC-08 | App MUST provide the same P0 record flows on iOS and Android browsers. |
| AC-09 | App MUST remain fully usable when Tencent Cloud base URL is empty, invalid, or unreachable. |
| AC-10 | MVP MUST optimize for one-family private use before multi-family sharing. |
| AC-11 | App MUST request persistent storage when available and display the result. |
| AC-12 | App MUST compress image attachments before local save by default. |
| AC-13 | Entities MUST include sync-ready metadata before backend integration. |

## Non-Goals

| Non-Goal | Reason |
|---|---|
| App Store native app | 当前目标是 Windows 可开发、iOS/Android 可用的 PWA |
| 社区/论坛 | 家庭自用，不做内容社区 |
| 电商/母婴推荐 | 与核心记录目标无关 |
| AI 医疗诊断 | 风险高，且不符合家庭记录工具定位 |
| 第一版多人实时协作 | 服务器暂未准备，先保留接口 |
| 第一版同事多家庭使用 | 自家稳定使用优先，小范围推荐必须等数据隔离和权限模型完成 |
| 第一版全自动 OCR 入库 | 必须人工确认，避免识别错误 |
